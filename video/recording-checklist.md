# Runwiser Recording Checklist

## Recording Rules

- Record short segments, not one continuous walkthrough.
- Leave `0.5s-1.0s` of clean lead-in and tail room on every clip.
- Keep cursor motion deliberate and slow.
- Prefer one focused scroll over multiple exploratory scrolls.
- Stop and re-record immediately if the page jitters, overscrolls, or reveals irrelevant content.
- Record at full desktop resolution so we can crop in post without losing clarity.

## Capture Order

### Scene 1: Hook

- Source: landing page hero
- Capture target: `6-8s`
- Action:
  - load hero fully
  - gentle pointer move
  - no scrolling
- Goal:
  - clear opening on title and positioning

### Scene 2: What Runwiser Is

- Source:
  - landing page positioning section
  - README top definition block
- Capture target: `10-12s`
- Action:
  - one short downward scroll on landing page
  - one clean cut to README
- Goal:
  - establish what the project is in plain language

### Scene 3: Why Proof Comes First

- Source:
  - README proof-first passages
  - optional supporting doc snippet
- Capture target: `12-14s`
- Action:
  - hold long enough for overlay placement
  - no long scrolls
- Goal:
  - support the proof-first animation insert

### Scene 4: PI Agent Orchestration

- Source:
  - workflow or architecture-related doc surfaces
  - any multi-agent workflow reference worth showing
- Capture target: `16-18s`
- Action:
  - brief reveal of the source material
  - leave room for the orchestration overlay animation
- Goal:
  - visually support the multi-agent explanation

### Scene 5: Skill Wiki Surface

- Source:
  - landing page wiki-related section only
- Capture target: `8-10s`
- Action:
  - short downward scroll
  - one hold on the section that frames the wiki in the overall product story
- Goal:
  - show the wiki as part of the public product surface

### Scene 6: Live Wiki Service

- Source:
  - real standalone wiki service
  - one catalog or listing page
  - one concrete skill or category detail page
- Capture target: `12-14s`
- Action:
  - open the real wiki service, not the landing page section
  - hold on the wiki home or catalog
  - click into a catalog or listing surface
  - click into a concrete detail page
  - perform one short scroll on the detail page
  - pause on the final page before cutting away
- Goal:
  - prove the wiki is a real service, not just a landing page concept
  - show an actual `catalog -> detail` interaction path

### Scene 7: Repo Architecture

- Source:
  - repo tree
  - `docs/architecture.md`
- Capture target: `14-16s`
- Action:
  - one focused reveal of key directories
  - one hold on the architecture doc
- Goal:
  - make the layered structure legible

### Scene 8: Proof In Practice

- Source:
  - one `runs/` example
  - one workflow asset
  - one KB asset
- Capture target: `12-14s`
- Action:
  - three short clips or one tightly segmented repo walkthrough
- Goal:
  - prove the repository is operational, not conceptual

### Scene 9: Close

- Source:
  - landing page hero or README summary area
- Capture target: `6-8s`
- Action:
  - clean static hold
- Goal:
  - leave space for the final value statement

## Output Naming

Use filenames like:

- `video/assets/raw/landing/scene-01-hero.mov`
- `video/assets/raw/landing/scene-02-positioning.mov`
- `video/assets/raw/wiki/scene-05-wiki-home.mov`
- `video/assets/raw/repo/scene-06-architecture.mov`
- `video/assets/raw/repo/scene-07-proof-path.mov`

## Must-Have Visuals

- landing page hero
- one clear proof-first message
- one PI-agent-orchestration support shot
- one landing-page wiki teaser shot
- one real wiki service `catalog -> detail` shot
- one repo tree shot showing:
  - `runs/`
  - `workflows/`
  - `workflow-kb/`
  - `scenarios/`

## Nice-To-Have Visuals

- a gentle zoom target on the five-layer architecture diagram
- a clean README crop that states the repository is not a Xiaohongshu tool
- a close crop on one representative proof asset in `runs/`
