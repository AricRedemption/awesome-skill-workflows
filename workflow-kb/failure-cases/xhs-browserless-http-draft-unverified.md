# XHS Browserless HTTP Draft Unverified

## Status

- Type: failure / unknown capability boundary
- Scenario: Xiaohongshu creator workflow
- Promotion status: not verified, not supported

## Boundary

`browserless_http` means creating or publishing a Xiaohongshu draft by sending direct HTTP requests without launching any browser engine.

This is different from `headless_browser`, where Chromium still runs in the background without a visible window.

## Current Finding

The project has evidence for `headless_browser` draft-save using MCP cookies and Puppeteer.

The project does not have verified evidence for `browserless_http` draft creation.

## Why It Is Not Supported Yet

Direct HTTP draft creation may require page-derived state that is not captured by a static cookie alone, such as:

- upload flow state,
- dynamic tokens,
- creator-page JavaScript state,
- request ordering,
- anti-abuse checks,
- draft metadata generated by the web UI,
- session state bound to the browser context.

Trying to bypass the creator UI with hand-written private API calls also increases platform-risk and compliance ambiguity.

## Required Support Path

Only consider `browserless_http` support if a future task can prove all of these without storing sensitive data:

1. The HTTP endpoint behavior is available through a documented, permitted, or otherwise scenario-approved interface.
2. Cookie/session handling is stored outside the repository and never written to logs or KB.
3. Upload, draft creation, and draft verification are separated into explicit steps.
4. The draft-only action is tested before any live publish path.
5. Human review and risk approval happen before account-state-changing calls.
6. Proof verifies the created draft through an independent read path.
7. The run is first stored as experimental or failed evidence, not a verified workflow.

## Prevention Rule

Do not describe this project as supporting Xiaohongshu browserless HTTP draft or publish workflows until verified evidence exists.
