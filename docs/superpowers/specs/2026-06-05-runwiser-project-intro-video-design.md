# Runwiser Project Intro Video Design

## Goal

Produce a polished English `16:9` project introduction video for
`awesome-skill-workflows` in under two minutes.

The video should explain three things clearly:

1. what Runwiser is,
2. why proof-first workflow execution matters,
3. how PI-style multi-agent orchestration fits into the system.

## Audience

- technical founders
- AI builders
- agent workflow developers
- open-source users evaluating whether the repository is real or only a concept

## Creative Direction

- Tone: clean technical
- Structure: chapter-based explainer
- Language: English only
- Voice: Xiaomi MiMo `Mia`
- Delivery style: natural, warm, technical, confident, not salesy

## Runtime Target

- Preferred range: `90-110s`
- Hard cap: `120s`

## Core Story

This repository is not a prompt collection or a one-off automation demo. It is
an evidence-first framework for turning repeated agent work into verified,
reusable workflow assets.

The video should carry two parallel messages:

1. proof comes before reuse,
2. multiple agents are useful only when their work is orchestrated, gated, and
   preserved as evidence.

## Visual Mix

- `70%` real screen capture
- `20%` overlays and guided highlights
- `10%` lightweight architecture motion

## Visual Sources

The video should combine all three of these surfaces:

1. landing page for first impression and positioning
2. skill wiki for discoverability and reusable asset surface
3. repo/docs for proof and implementation depth

## Chapter Structure

1. Hook
2. What Runwiser Is
3. Why Proof Comes First
4. PI Agent Orchestration
5. Skill Wiki Surface
6. Repo Architecture
7. Proof In Practice
8. Close

## Motion Strategy

Motion should explain structure, not distract from it.

Use animation mainly for:

- layer transitions
- gated workflow handoffs
- highlighting repo paths
- showing run -> KB -> verified asset progression

Avoid flashy transitions, excessive camera movement, or dense on-screen text.

## Production Notes

- Record in short segments, not one continuous take.
- Use narration timing as the master timeline.
- Keep every scene focused on one idea.
- Introduce a visible state change every `6-10s`.
- Prefer zooms, callouts, and focus crops over heavy motion graphics.

## Deliverables

- `video/script.md`
- `video/storyboard.md`
- segmented TTS audio files
- final `16:9` MP4
